## 架构总览

ScadGenerator 采用**多智能体协作架构**，核心设计理念：

```
┌──────────────────────────────────────────────────────────┐
│                    系统核心价值                           │
│  ✓ 不是「单模型一次性回答」                               │
│  ✓ 而是「角色分工 + 路由分流 + 可恢复会话」             │
│  ✓ 通过智能体间协作提升最终成功率                       │
└──────────────────────────────────────────────────────────┘
```

### 架构层次

```
┌─────────────────────────────────────────────────────────┐
│ 业务层 (3个角色)                                          │

## 三角色分工模型

### 角色定义与职责

#### 1. 产品经理 (Product Manager)

**身份**: 小K - 可爱、专业的3D参数化建模PM  
**核心职责**:
- 需求澄清与多轮确认
- 将模糊需求结构化
- 生成标准化的建模方案

**工作方式**:
- **对话模式**: 多轮交互，逐步引导用户明确需求
- **输出格式**: 结构化文本（含【问题】【反馈】【状态】标记）
- **约束**: 不输出任何代码、JSON或可执行内容

**职责边界**:
```
✓ 做什么:
  ├─ "请问尺寸大概多少毫米？"
  ├─ "是否需要孔洞或倒角？"
  └─ "确认需求: 200×100×50mm的参数化立方体"

✗ 不做什么:
  ├─ 不输出OpenSCAD代码
  ├─ 不输出JSON或其他代码格式
  └─ 不进行技术问题诊断
```

**模型与API**:
| 参数 | 值 |
|---|---|
| 模型 | PM_MODEL (moonshotai/kimi-k2.5) |
| Protocol | OpenAI Compatible 或 Anthropic Messages |
| Temperature | 0.5 (对话风格) |
| Max Tokens | 2048 |
| 超时 | 120s |

---

#### 2. 老师傅 (Master Craftsman)

**身份**: 经验丰富的OpenSCAD代码架构师  
**核心职责**:
- 最终OpenSCAD代码生成
- 疑难问题升级处理
- 代码质量把控

**工作方式**:
- **代码模式**: 直接输出最终可执行代码
- **输出格式**: 纯SCAD源码，无任何解释或注释
- **约束**: 禁止markdown围栏、禁止多次输出同一代码

**职责边界**:
```
✓ 做什么:
  ├─ 从需求生成参数化OpenSCAD代码
  ├─ 处理复杂的几何组合问题
  ├─ 代码审查与修复（当反复失败时）
  └─ 参数化方案设计

✗ 不做什么:
  ├─ 不输出代码的解释或说明
  ├─ 不尝试第一次简单修复（交给实习生）
  └─ 不生成markdown格式的代码块
```

**模型与API**:
| 参数 | 值 |
|---|---|
| 模型 | OPENSCAD_MODEL (claude-4.5-sonnet) |
| 优先级读取 | QINIU_DEEPSEEK_MODEL → OPENSCAD_MODEL → claude-4.5-sonnet |
| Temperature | 0.2 (代码稳定) 或 0.7 (初次生成) |
| Max Tokens | 1536 (确认时) 或 4096 (生成时) |
| 超时 | 120s (确认) 或 240s (修复) |

---

#### 3. 实习生 (Intern)

**身份**: 勤快、细心的代码修复员  
**核心职责**:
- 编译错误自动修复
- 代码问题咨询处理
- 最小修改原则修复

**工作方式**:
- **修复模式**: 直接输出修复后代码
- **输出格式**: 纯SCAD源码，保留原有参数和逻辑
- **约束**: 禁止重大重构，优先最小改动

**职责边界**:
```
✓ 做什么:
  ├─ 修复编译错误（语法、缺失括号等）
  ├─ 保留原有建模意图与参数命名
  ├─ 做最小化修改使其可编译
  └─ 处理首次代码问题

✗ 不做什么:
  ├─ 不进行大规模重构
  ├─ 不改变参数名称（除非必要）
  ├─ 不处理疑难问题（升级给老师傅）
  └─ 不输出代码解释
```

**模型与API**:
| 参数 | 值 |
|---|---|
| 模型 | INTERN_MODEL (deepseek/deepseek-v3.2-251201) |
| 优先级读取 | INTERN_MODEL → QINIU_DEEPSEEK_MODEL → deepseek/deepseek-v3.2-251201 |
| Temperature | 0.2 (修复稳定) |
| Max Tokens | 1536 |
| 超时 | 120s (确认) 或 240s (修复) |

### 选型理由

#### Claude 4.5 为何是老师傅首选

```
优势:
  ✓ 代码质量最高（特别是复杂结构）
  ✓ 参数化设计能力强
  ✓ 错误率较低（首次生成成功率高）
  ✓ 支持Anthropic Messages协议（兼容性好）

劣势:
  ✗ 相对较慢（30-60s）
  ✗ 成本较高

使用场景:
  ✓ 首次生成（质量最重要）
  ✓ 疑难问题升级处理
  ✓ 反复失败后的最后手段
```

#### DeepSeek v3.2 为何是实习生首选

```
优势:
  ✓ 修复速度快（15-30s）
  ✓ 成本低
  ✓ 修复错误类型丰富
  ✓ 语言模型能力好（理解代码上下文）

劣势:
  ✗ 初次生成成功率略低
  ✗ 大型复杂结构支持不如Claude

使用场景:
  ✓ 编译错误快速修复（首次）
  ✓ 语法/格式问题处理
  ✓ 参数错误修复
```

#### Kimi k2.5 为何是双重角色

```
优势:
  ✓ 对话能力强（最适合需求澄清）
  ✓ 理解能力好（能从错误恢复代码）
  ✓ 通用性强（修复兜底救援）
  ✓ 量身定制不复杂需求

劣势:
  ✗ 初次生成不及Claude
  ✗ 速度不如DeepSeek

使用场景:
  ✓ 产品经理需求澄清（主职）
  ✓ 结构化方案生成
  ✓ DeepSeek失败的二次修复（兜底）
```

### 模型读取优先级链

系统使用**分层优先级策略**确保首选模型可用，同时提供兜底方案。

#### 老师傅模型优先级

```
实际使用的老师傅模型 = 
  QINIU_DEEPSEEK_MODEL (若设置)
  ↓ 未设置则读取
  OPENSCAD_MODEL (若设置)
  ↓ 未设置则使用
  'claude-4.5-sonnet' (默认)

配置示例:
# .env
QINIU_DEEPSEEK_MODEL=claude-4.5-sonnet   # ← 优先级最高
OPENSCAD_MODEL=gpt-4-turbo               # ← 次优先级

```

#### 实习生模型优先级

```
实际使用的实习生模型 = 
  INTERN_MODEL (若设置)

  ↓ 未设置则使用
  'deepseek/deepseek-v3.2-251201' (默认)

配置示例:
# .env
INTERN_MODEL=gpt-3.5-turbo               # ← 优先级最高

```

配置示例:
# .env
PM_MODEL=moonshotai/kimi-k2.5  # ← 明确指定

### 核心提示词详解

#### 4.1 老师傅：代码生成提示词

**模板版本** (用于初次生成与代码问题处理):

```
你是"老师傅"，负责 OpenSCAD 代码生成。
只输出一段可执行的 OpenSCAD 代码，禁止输出任何额外文本。

强制规则（必须遵守）：
1) 禁止解释、分析、思考过程、提示词复述。
2) 禁止 markdown 代码围栏（例如 ```openscad）。
3) 禁止返回重复代码块，只允许一段最终代码。
4) 生成有效且可编译的 OpenSCAD。
5) 尽量参数化（使用顶层参数定义）。

输出要求：
- 只返回纯 OpenSCAD 源码，不要前后缀。
```

**参数化版本** (用于确认链路):

```
你是"${roleName}"，负责直接修改并输出最终 OpenSCAD 代码。
输出要求与老师傅代码生成一致。

强制规则（必须遵守）：
1) 禁止解释、分析、思考过程、提示词复述。
2) 禁止 markdown 代码围栏（例如 `openscad）。
3) 禁止返回重复代码块，只允许一段最终代码。
4) 生成有效且可编译的 OpenSCAD。
5) 尽量参数化（使用顶层参数定义），并保留原有建模意图与参数命名。
```

**关键特性分析**:

| 特性 | 作用 | 实现方式 |
|---|---|---|
| 角色确认 | 让模型明确身份 | "你是老师傅" |
| 输出约束 | 防止冗余文本 | "禁止解释/禁止markdown" |
| 参数化强调 | 驱动参数提取 | 强制规则第5条 |
| 编译性保证 | 降低修复成本 | "生成有效且可编译的" |

#### 4.3 产品经理：需求澄清提示词

```
你是小K，一位可爱又专业的 3D 参数化建模产品经理~ ✨

你的职责是帮用户把模糊的建模想法变成清晰的需求，
然后交给老师傅去写代码！

!! 重要 !!：这是关于用代码生成 3D CAD 模型的，不是网页设计或其他类型的项目。
!! 强制规则 !!：你绝对不能输出 OpenSCAD 代码、JSON、代码块或任何可执行代码。
你只负责需求确认哦~

你的任务：
1) 当用户描述想要的 3D 模型后，主动询问关键细节（像聊天一样自然）
2) 通过多轮对话逐步明确需求，语气要亲切可爱~
3) 当信息足够时输出【需求确认完成】并给出【最终需求】
4) 只有当用户明确说"生成代码/开始生成/出代码"时，
   才输出【请老师傅生成代码】

需要确认的关键信息：
✓ 主体几何形状（立方体、圆柱、球体、锥体或组合）
✓ 关键尺寸参数（长、宽、高或半径等，单位毫米 mm）
✓ 需要参数化的变量（哪些尺寸是可调的）
✓ 特殊特征（孔洞、倒角、圆角、凹陷等）
✓ 组合方式（并集、差集、交集）

回复格式范例：
【问题】
- 嗨~ 想做什么样的模型呀？立方体还是其他形状呢？
- 尺寸大概多少毫米呀？📏

【反馈】
好哒！我理解的是一个 200×100×50mm 的参数化立方体，是不是这样呀？

信息完整时请输出：
【需求确认完成】
【最终需求】
- 用 4~8 条要点总结可用于老师傅生成代码的完整需求

如果用户还没说"生成代码"，请额外输出：
【状态】等待用户下达"生成代码"指令

只有当用户明确要求生成代码时，再额外输出：
【请老师傅生成代码】

语气：可爱、亲切、活泼，像一位耐心的产品经理小姐姐~ 
可以适度使用 emoji 但不要太多哦！
```

**对话标记设计**:

```
【问题】→ 产品经理需要什么
【反馈】→ 确认理解
【需求确认完成】→ 阶段完成标志
【最终需求】→ 标准化需求输出
【状态】→ 等待下一步指令
【请老师傅生成代码】→ 触发代码生成

后端通过正则匹配这些标记来判断对话进度:
- isClear = response.includes('【需求确认完成】')
- shouldGenerate = response.includes('【请老师傅生成代码】')
- confirmedRequirement = extractText('【最终需求】', '【状态】')
```

### 提示词优化最佳实践

#### 1. 约束强度级别

```
弱约束 (低效):
  "尽量不输出解释"
  
中等约束 (可用):
  "禁止输出代码解释"
  
强约束 (推荐):
  "强制规则：禁止解释、分析、思考过程、提示词复述"
  
超强约束 (关键功能):
  "强制规则（必须遵守）：
   1) 禁止...
   2) 禁止...
   3) ..."
```

#### 2. 参数化设计

```
不参数化 (硬编码):
  "你是老师傅"
  
参数化 (灵活):
  "你是\"${roleName}\"，负责..."
  
实现代码:
  const roleName = role === 'master' ? '老师傅' : '实习生';
  const systemPrompt = `你是"${roleName}"，负责...`;
```

#### 3. 输出格式标记

```
无标记 (难以解析):
  产品经理回复...
  
有标记 (易于处理):
  【需求确认完成】
  【最终需求】
  - 需求点1
  - 需求点2
  
后端处理:
  const isClear = response.includes('【需求确认完成】');
  const requirement = extractConfirmedRequirement(response);
```

## 性能与成本综合分析

### 三种部署策略

#### 策略A: 全质量优先 (推荐)

```
生成: Claude + Kimi        → 质量最高，成本最高
修复: DeepSeek → Kimi × 2  → 梯级兜底
特点: 适合to B企业应用
成本: 每次生成 $0.5-1.5
```

#### 策略B: 成本优先

```
生成: Kimi                  → 速度快，成本低
修复: DeepSeek → Kimi      → 双格尝试
特点: 适合大规模C端应用
成本: 每次生成 $0.1-0.3
```

#### 策略C: 混合平衡

```
生成: DeepSeek (首次) 
      → Claude (反复失败)  → 成本和质量平衡
修复: DeepSeek → Kimi      → 
特点: 适合经费有限的团队
成本: 每次生成 $0.2-0.8
```
